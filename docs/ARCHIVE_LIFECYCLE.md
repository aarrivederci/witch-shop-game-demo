# 归档生命周期策略

> 为防止 `archives/` 无限膨胀又不丢失回退能力，本文件给出归档保留 / 折叠 / 删除规则。

## 1. 归档分类

| 类型 | 命名后缀 | 何时创建 | 保留策略 |
|---|---|---|---|
| 大版本 baseline 完整快照 | `<version>-baseline-<ts>-full` | 每个 `vX.Y.0` 封版后 | **永久保留** |
| 小版本完整快照 | `<version>-<topic>-<ts>-full` | 大改动（多文件 / 数据结构 / 核心 UI）前 | 大版本封版后可折叠为最新一份 |
| 修改前轻量归档 | `<version>-<topic>-pre-<ts>` | 每次修改前必做 | 同一小版本的 pre 系列在该小版本封版后可折叠为最末一份 |
| 后期附加归档 | 其他名称 | 实验、补救、QA 截图 | 视情况保留或折叠 |
| `.DIRTY` 后缀归档 | `*-<ts>.DIRTY` | post-flight 失败后自动重命名 | 修复后人工删除 |

## 2. 折叠规则

当 `vX.Y` 已封版进入 `vX.Y+1`：

1. 找出 `vX.Y.*-pre-*` 所有轻量归档。
2. 保留最早一份（封版起点）与最末一份（封版终点）。
3. 中间归档移到 `archives/_folded/vX.Y/` 子目录。
4. 在 `archives/_folded/vX.Y/README.md` 列出原始名称、`manifest.json` 的 goal 字段。

折叠是**移动而不是删除**。仅 `.DIRTY` 归档与显式判定为重复的归档才可删除。

## 3. 删除允许的情况

- post-flight 失败的 `.DIRTY` 归档，且对应修改未保留。
- 用户明确要求删除某个归档。
- 同一时间戳分钟内被多次创建产生的重名残骸。

## 4. 全量 baseline 重建

如果归档目录损坏或想从 0 重建：

1. 找最近一个 `*-baseline-*-full` 完整快照。
2. 复制其内容到 `witch-shop-game/`。
3. 跑 `node tools/project_check.js` 确认可运行。
4. 在 `CHANGELOG.md` 写一条 "归档目录重建于 \<baseline-name\>"。

## 5. 工具支持

当前**没有**自动折叠脚本。建议在 NEXT.md 中开 backlog 写一个 `tools/fold_archives.js`，在小版本封版时调用。

## 6. 当前归档量级

- 总数：约 48 个目录（v3.3.23 之前）。
- 完整快照：`v3.1-baseline-20260614-0428-full` 1 个。
- 其余均为轻量归档。

体积没有立即压力，但**禁止在不读本文件的情况下批量删除任何 `archives/` 内容**。
