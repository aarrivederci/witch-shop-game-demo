# Witch Shop Game v3.0 路线图

本文档用于把 3.0 大版本拆分为可执行的小版本计划。`STORY_SYSTEM_ROADMAP.md` 负责长期设计规范；本文档负责 3.0 阶段的实际开发边界、顺序与验收标准。

## 1. 版本定位

### v2.0 基线

上一个稳定大版本，可理解为：基础经营、订单、模块解锁、故事阅读与羁绊雏形已经存在。

### v3.0 总目标

将游戏从“扁平经营 + 解锁后阅读故事”升级为一个互相反馈的经营叙事循环：

```text
订单表现 → 金币产出/消耗 → 商品解锁/礼物/店铺投资 → 羁绊变化
→ 特殊 NPC 访问反馈/故事解锁/回馈助力 → 新订单/新反馈
```

核心解决三个问题：

1. 金币后期缺少持续用途。
2. 商品、故事解锁与经营、羁绊、礼物、订单表现关联不足。
3. 角色目前更多存在于故事阅读页，尚未充分进入订单经营主循环。

同时为后续多语言词库、自定义 NPC、自定义内容包预留结构。

### v3.0 优化后的系统框架（2026-06-15）

v3.0 阶段的重心不再只是“增加金币消耗项”或“扩写故事条件”，而是把商品、NPC 与经营循环打通：

```text
商品系统
  ├─ 商品用于订单与模块成长
  ├─ 新商品解锁时展示介绍与用途
  ├─ 已解锁商品可作为礼物赠送
  └─ 商品标签用于匹配特殊 NPC 偏好

专精系统
  ├─ 模块专精来自长期经营
  ├─ 专精影响对应模块商品的礼品品质
  └─ 礼品品质影响羁绊收益与回馈触发强度

NPC 系统
  ├─ 一般 NPC：性格 + 身份组合，负责稳定订单流与世界氛围
  ├─ 特殊 NPC：已有 20 个具名角色，负责关系经营与长期记忆
  ├─ 羁绊影响特殊 NPC 来店台词、称呼、订单表现与后续故事
  └─ 特殊 NPC 可拥有商品偏好、订单偏好、身份助力与专属反馈

订单系统
  ├─ 一般 NPC 保证基础订单刷新
  ├─ 特殊 NPC 以一定概率作为熟客来访
  ├─ 特殊 NPC 订单卡显示具名身份与羁绊等级
  └─ 后续可扩展特殊委托、异常订单与里世界入口
```

一句话原则：

```text
普通客人让小店热闹，特殊客人让小店有记忆。
```

羁绊的作用也相应扩展为：

```text
解锁故事 + 改变来访台词 + 影响特殊订单/回馈 + 体现 NPC 与小店关系变化
```

## 2. 小版本拆分

## v3.1：版本整理与系统基线

### 目标

把当前项目整理为 3.0 开发基线，避免后续功能混杂推进。

### 范围

- 修正版本命名：当前 `v0.2.0` 语义调整为 `v3.1-baseline`。
- 完善 `CHANGELOG.md` 与 `docs/VERSION_ARCHIVE.md`。
- 保留已有归档目录，但在文档中说明其历史命名原因。
- 整合 `STORY_SYSTEM_ROADMAP.md`，补充多语言词库规划。
- 新增 `AI_HANDOFF.md`，作为详细协作交接说明。
- 新增 `START_HERE.md`，作为用户和后续模型的唯一记忆入口。
- 新增 `docs/DATA_CONTRACTS.md`，明确数据字段、ID 与存档兼容契约。
- 新增 `docs/QA_CHECKLIST.md`，补充语法检查之外的手动验证清单。
- 创建完整可运行基线快照与轻量协作文档归档。
- 确认当前语法检查通过。

### 验收标准

- 文档中明确 `v2.0 → v3.0` 的版本关系。
- 3.0 小版本拆分清晰。
- 用户只需要记住 `START_HERE.md`，后续模型/协作者可从该入口接手。
- 后续模型/协作者能通过 `AI_HANDOFF.md`、`CHANGELOG.md`、`docs/V3_ROADMAP.md` 判断当前阶段。
- 数据结构修改前有 `docs/DATA_CONTRACTS.md` 可参考。
- 修改后有 `docs/QA_CHECKLIST.md` 可验证。

## v3.1-automation：归档与检查自动化

### 目标

降低多模型协作时用户的记忆负担，把归档、检查、交接提示从“靠人记住”改为“由脚本和入口文档承接”。

### 主要功能

- 新增 `tools/create_archive.js`：自动创建轻量归档或完整快照，并生成 `MANIFEST.md`。
- 新增 `tools/project_check.js`：统一运行语法检查、关键文档检查、归档结构检查。
- 可选新增 `tools/handoff_status.js`：输出可直接复制给后续模型的接手提示。
- 更新 `AI_HANDOFF.md`、`docs/VERSION_ARCHIVE.md`，把自动化流程写入规范。

### 验收标准

- 用户不需要手动记忆归档目录结构。
- 新模型接手时只需要读取 `START_HERE.md`。
- 修改前可以用一条命令创建归档。
- 修改后可以用一条命令完成基础检查。

## v3.2：商品赠礼与特殊 NPC 关系循环

### 目标

在已经完成店铺投资、金币消费统计与礼物故事条件样例的基础上，把 v3.2 从“单纯补金币消耗出口”升级为：

```text
商品解锁 → 商品介绍/样品 → 商品赠礼 → 羁绊变化
→ 特殊 NPC 来访台词变化/回馈提示 → 后续故事或订单入口
```

v3.2 的核心不是一次性做完完整角色系统，而是建立一个最小可运行闭环，让玩家能明确感到：

```text
经营获得的商品与金币，会改变自己和具名 NPC 的关系；
羁绊提高后，NPC 不只解锁故事，也会在订单来访时以不同台词回应玩家。
```

详细落地拆解见：

```text
docs/V3_2_PRODUCT_GIFT_SPECIAL_NPC_PLAN.md
```

### 主要功能

- 商品介绍与样品：新商品解锁时展示简短介绍，并可获得少量赠礼样品或赠礼资格。
- 商品赠礼：已解锁商品可作为礼物赠送给特殊 NPC，商品标签与 NPC 偏好决定羁绊收益。
- 专精影响礼品品质：模块专精越高，对应商品作为礼物时品质越高，羁绊收益或回馈概率越好。
- 一般 NPC / 特殊 NPC 分层：
  - 一般 NPC：由“性格 + 身份”组合生成，例如“温柔的旅人”“急躁的商贩”。
  - 特殊 NPC：已有 20 个具名角色，进入熟客池，拥有专属台词、偏好与羁绊表现。
- 特殊 NPC 来访台词：订单生成抽中特殊 NPC 时，根据当前羁绊等级读取不同 `visitLines`。
- 特殊 NPC 订单卡表现：显示角色名、身份/称号、羁绊等级或熟客标记。
- 羁绊阶段回馈提示：羁绊达到阶段后，至少展示一次“该 NPC 提供了什么帮助/回馈”的明确提示；第一版可先记录与展示，不强制接入复杂效果。
- 兼容既有金币循环：保留店铺投资、金币消费统计与礼物故事条件，不推翻已完成的 v3.2 工作。

### 当前状态与边界校准（2026-06-14）

当前 v3.2 已经完成或接近完成的部分：

- 店铺投资与店铺气质：已接入投资页签、`SHOP_INVESTMENTS`、`shopTraits`、`shopInvestments`，并提供少量店铺气质故事闭环。
- 金币消费统计：已接入 `coinSpendStats` 与 `spendCoins()` 分类统计，可区分礼物、羁绊、店铺投资等消费方向。
- 礼物系统深化：已支持送礼统计、指定礼物计数、重复赠礼衰减、`unlockGifts` 与故事/羁绊页目标提示。
- 故事提示补强：已补齐店铺气质、消费统计、送礼条件等目标摘要，作为 v3.2 的少量反馈样例。

需要特别注意：这些故事相关改动只是为了验证“金币选择会改变故事线索”的最小闭环，不代表 v3.2 要继续扩展完整故事条件网络。

### 路线再校准（2026-06-15）

在 2026-06-15 的框架讨论后，v3.2 后续不再优先补“经营消耗品 / 风险控制”作为唯一收尾目标，而是先把商品、赠礼、特殊 NPC 与订单来访打通。原因是：

```text
经营消耗品解决金币出口；
商品赠礼与特殊 NPC 解决经营行为如何被角色记住。
```

因此 v3.2 拆成以下实施层级：

```text
v3.2.1 文档与数据契约整理
  - 明确商品、礼物、NPC 分层、羁绊台词、回馈字段。
  - 判断旧 BOND_GIFTS 与商品赠礼的兼容策略。

v3.2.2 商品介绍与赠礼样品最小闭环
  - 商品解锁时有介绍。
  - 已解锁商品可进入赠礼选择。
  - 商品标签影响特殊 NPC 羁绊收益。

v3.2.3 特殊 NPC 来访最小闭环
  - 订单顾客区分一般 NPC 与特殊 NPC。
  - 特殊 NPC 显示具名身份与羁绊等级。
  - 特殊 NPC 台词随羁绊等级变化。

v3.2.4 羁绊回馈提示与版本收尾
  - 羁绊阶段奖励先以提示/记录形式落地。
  - 补充 QA 清单与数据契约。
  - 根据内容量决定是否把经营消耗品顺延为 v3.2.5 或并入后续经济迭代。
```

v3.2 应继续暂缓以下内容到 v3.3 或更后：

- 统一 `requirements` 字段结构。
- 大规模新增经营统计、流失顾客、订单表现等故事条件。
- 半隐藏预告系统重构。
- 表世界 / 里世界 / IF 故事网络扩写。
- 流失订单作为叙事入口的完整实现。
- 每个特殊 NPC 的完整专属订单链。
- 每个商品的 NPC 专属台词。
- 身份助力的复杂数值效果。

### 原金币消耗目标的保留位置

经营消耗品 / 风险控制仍然有价值，但不再作为当前最优先落地点。建议处理方式：

```text
如果 v3.2.2-v3.2.4 内容量可控：作为 v3.2.5 追加 1-2 个一次性道具。
如果商品/NPC 闭环内容量较大：顺延到 v3.3 前置或独立经济小版本。
```

### 设计原则

- 金币消耗要提供“方向选择”，而不是只做数值黑洞。
- 礼物和商品用于“靠近角色”，店铺投资用于“改变店铺身份”，消耗品用于“改变经营节奏”。
- 一般 NPC 负责订单流量，特殊 NPC 负责关系记忆，二者不能互相取代。
- 羁绊首先影响玩家可见反馈：来访台词、称呼、订单卡表现、回馈提示。
- 特殊 NPC 是增强体验，不应阻塞基础订单刷新。
- 中后期金币增长可以保留，但应让玩家始终有值得等待、规划和取舍的目标。
- 不把流失订单简单定义为口碑惩罚；它也可以是里世界、异常顾客、隐藏委托的入口。

### 验收标准

- 玩家在中后期有稳定金币消耗或商品消耗出口。
- 至少 2-3 类资源用途不是单纯数值膨胀，而能影响羁绊、NPC 反馈或故事。
- 不破坏原有订单收益和模块解锁流程。
- 玩家可以通过商品赠礼主动推进至少一类角色关系线索。
- 至少存在一个“商品赠礼 → 羁绊提升 → 特殊 NPC 来访台词变化”的闭环样例。
- 至少存在一个“金币消费/店铺投资 → 故事预告变化/故事可解锁”的既有闭环样例继续可用。

## v3.3：故事条件网络、特殊委托与半隐藏预告

### 目标

让故事从线性解锁变为与经营数据、角色羁绊、商品赠礼、金币消费、特殊 NPC 来访和世界状态相关的条件网络。

v3.3 的重点是让故事不再只是“满足数值后出现”，而是让玩家通过半隐藏线索理解：自己经营店铺的方式、把金币花在哪里、如何对待流失订单，正在把世界推向不同叙事入口。

### 主要功能

- 统一故事字段：`scope`、`route`、`characterIds`、`preview`、`requirements`。
- 支持更多 `requirements`：羁绊、商品赠礼、礼物标签、经营统计、流失顾客、金币消费、特殊 NPC 访问/回馈记录。
- 锁定故事显示半隐藏式预告。
- 羁绊页显示对应角色的可推进故事线索。
- 特殊 NPC 高羁绊后可出现特殊委托或异常访问。
- 增加若干经营 IF 故事，例如异常委托、贵族顾客、被冷落的熟客、夜间来客。
- 建立“表世界 / 里世界 / 店铺线”三类推进入口：
  - 表世界：常规订单、常规礼物、稳定羁绊。
  - 里世界：高羁绊、主题礼物、反常订单、流失顾客或特殊投资。
  - 店铺线：累计消费、店铺投资方向、订单结构与顾客反馈。
- 流失订单作为叙事入口：高流失不必只触发口碑危机，也可以触发异常顾客、隐藏委托、角色试探或世界观裂缝。

### 验收标准

- 玩家能通过线索理解故事大致推进方向。
- 至少存在角色故事、店铺故事、IF 故事三种类型。
- 经营失败不只是惩罚，也能产生叙事反馈。
- 至少存在一个 `inner` 故事由“金币消费/主题礼物/流失订单”共同影响。
- 锁定故事的 `preview.visibleHints` 能解释玩家应该从经营、送礼或店铺投资哪个方向尝试推进。

## v3.4：内置多语言词库包系统

### 目标

把打字系统从固定英语词库升级为可选择的内置词库包系统，同时保持网页发布开箱即玩。

### 主要功能

- 新增词库包格式：`packId / language / inputMode / entries`。
- 内置英语词库包。
- 内置小型日文幻想主题词库包，采用 `romaji` 输入。
- 订单系统支持 `input / display / meaning`。
- 设置当前词库，默认仍为英语。
- 加载失败时回退到基础内置词库。

### 暂不做

- 玩家自定义词库导入。
- 原生日文假名输入。
- URL 远程词库加载。

### 验收标准

- 网页发布版本不导入任何外部文件也能正常游玩。
- 英语旧词库兼容。
- 日文词库以“显示日文 + 输入罗马音”的方式可玩。

---

## v3.5：自定义词库导入与本地保存

### 目标

让玩家可以在网页中导入自己的 JSON 词库，并在本地保存。

### 主要功能

- 文件选择器导入 `.json` 词库。
- 词库格式校验。
- 保存到 `localStorage`，大词库后续再考虑 `IndexedDB`。
- 删除已导入词库。
- 导出词库。
- 词库错误提示。

### 验收标准

- 导入错误不会破坏游戏。
- 清空导入词库后仍能回到内置词库。
- 自定义词库不作为网页发布的必要条件。

## v3.6：内容包与长期扩展预研

### 目标

为自定义 NPC、故事包、词库包统一导入/导出做准备。

### 主要功能

- 设计内容包结构：`characters / gifts / stories / relationships / wordPacks`。
- 冲突检测：ID、角色引用、礼物引用、故事条件可达性。
- 角色搜索、筛选、排序预研。
- 关系故事界面预研。

### 验收标准

- 有明确数据结构文档。
- 不急于做完整编辑器，先保证结构可扩展。

## 3. 当前推荐下一步

当前 **v3.1：版本整理与系统基线**、**v3.1-automation：归档与检查自动化** 已完成，项目正在推进 v3.2 运行时迭代。

推荐下一步优先做：

```text
v3.2.1 文档与数据契约整理
```

随后按内容量拆分推进：


```text
v3.2.2 商品介绍与赠礼样品最小闭环
→ v3.2.3 特殊 NPC 来访最小闭环
→ v3.2.4 羁绊回馈提示与版本收尾
→ 视内容量决定经营消耗品作为 v3.2.5 追加或顺延
→ v3.3 故事条件网络、特殊委托与半隐藏预告
→ v3.4 内置多语言词库包
→ v3.5 自定义词库导入
```

## 4. 范围控制原则

- 不在 v3.1 直接改大功能。
- 商品 / 金币 / 羁绊 / NPC 关系循环优先于多语言系统。
- 内置词库优先于导入词库。
- 日文先做罗马音输入，不做原生 IME 假名输入。
- 所有故事条件必须有玩家可理解的线索。
- 用户不需要记住复杂交接流程；后续模型应从 `START_HERE.md` 自行读取上下文。
- 每轮迭代前后都要更新 `CHANGELOG.md` 与归档说明。
