# Witch Shop Game v3.0 路线图

本文档用于把 3.0 大版本拆分为可执行的小版本计划。`STORY_SYSTEM_ROADMAP.md` 负责长期设计规范；本文档负责 3.0 阶段的实际开发边界、顺序与验收标准。

## 1. 版本定位

### v2.0 基线

上一个稳定大版本，可理解为：基础经营、订单、模块解锁、故事阅读与羁绊雏形已经存在。

### v3.0 总目标

将游戏从“扁平经营 + 解锁后阅读故事”升级为：

```text
订单表现 → 金币产出/消耗 → 礼物/店铺投资 → 羁绊变化 → 故事解锁 → 新订单/新反馈
```

核心解决两个问题：

1. 金币后期缺少持续用途。
2. 故事解锁与经营、羁绊、礼物、订单表现关联不足。

同时为后续多语言词库、自定义 NPC、自定义内容包预留结构。

## 2. 小版本拆分

## v3.1：版本整理与系统基线

### 目标

把当前项目整理为 3.0 开发基线，避免后续功能混杂推进。

### 范围

- 修正版本命名：当前 `v0.2.0` 语义调整为 `v3.1-baseline`。
- 完善 `CHANGELOG.md` 与 `docs/VERSION_ARCHIVE.md`。
- 保留已有归档目录，但在文档中说明其历史命名原因。
- 整合 `STORY_SYSTEM_ROADMAP.md`，补充多语言词库规划。
- 新增 `AI_HANDOFF.md`，作为详细协作交接说明。
- 新增 `START_HERE.md`，作为用户和后续模型的唯一记忆入口。
- 新增 `docs/DATA_CONTRACTS.md`，明确数据字段、ID 与存档兼容契约。
- 新增 `docs/QA_CHECKLIST.md`，补充语法检查之外的手动验证清单。
- 创建完整可运行基线快照与轻量协作文档归档。
- 确认当前语法检查通过。

### 验收标准

- 文档中明确 `v2.0 → v3.0` 的版本关系。
- 3.0 小版本拆分清晰。
- 用户只需要记住 `START_HERE.md`，后续模型/协作者可从该入口接手。
- 后续模型/协作者能通过 `AI_HANDOFF.md`、`CHANGELOG.md`、`docs/V3_ROADMAP.md` 判断当前阶段。
- 数据结构修改前有 `docs/DATA_CONTRACTS.md` 可参考。
- 修改后有 `docs/QA_CHECKLIST.md` 可验证。

## v3.1-automation：归档与检查自动化

### 目标

降低多模型协作时用户的记忆负担，把归档、检查、交接提示从“靠人记住”改为“由脚本和入口文档承接”。

### 主要功能

- 新增 `tools/create_archive.js`：自动创建轻量归档或完整快照，并生成 `MANIFEST.md`。
- 新增 `tools/project_check.js`：统一运行语法检查、关键文档检查、归档结构检查。
- 可选新增 `tools/handoff_status.js`：输出可直接复制给后续模型的接手提示。
- 更新 `AI_HANDOFF.md`、`docs/VERSION_ARCHIVE.md`，把自动化流程写入规范。

### 验收标准

- 用户不需要手动记忆归档目录结构。
- 新模型接手时只需要读取 `START_HERE.md`。
- 修改前可以用一条命令创建归档。
- 修改后可以用一条命令完成基础检查。

---

## v3.2：金币消耗与羁绊经济循环

### 目标

解决金币后期只增长、不消耗的问题。

### 主要功能

- 礼物系统深化：礼物价格、稀有度、角色偏好、礼物标签。
- 店铺投资：装修、招牌、魔法灯、货架、占卜桌等金币消耗项。
- 经营消耗品：订单加速、失败保护、提高某类订单出现率等。
- 金币消费统计：记录总消费、礼物消费、店铺投资消费。
- 金币消费与故事条件联动，例如 `coinsSpentOnBondMin`。

### 验收标准

- 玩家在中后期有稳定金币消耗出口。
- 至少 2-3 类金币用途不是单纯数值膨胀，而能影响羁绊或故事。
- 不破坏原有订单收益和模块解锁流程。

---

## v3.3：故事条件网络与半隐藏预告

### 目标

让故事从线性解锁变为与经营数据、角色羁绊、礼物和世界状态相关的条件网络。

### 主要功能

- 统一故事字段：`scope`、`route`、`characterIds`、`preview`、`requirements`。
- 支持更多 `requirements`：羁绊、礼物、礼物标签、经营统计、流失顾客、金币消费。
- 锁定故事显示半隐藏式预告。
- 羁绊页显示对应角色的可推进故事线索。
- 增加若干经营 IF 故事，例如口碑危机、贵族顾客、被冷落的熟客。

### 验收标准

- 玩家能通过线索理解故事大致推进方向。
- 至少存在角色故事、店铺故事、IF 故事三种类型。
- 经营失败不只是惩罚，也能产生叙事反馈。

---

## v3.4：内置多语言词库包系统

### 目标

把打字系统从固定英语词库升级为可选择的内置词库包系统，同时保持网页发布开箱即玩。

### 主要功能

- 新增词库包格式：`packId / language / inputMode / entries`。
- 内置英语词库包。
- 内置小型日文幻想主题词库包，采用 `romaji` 输入。
- 订单系统支持 `input / display / meaning`。
- 设置当前词库，默认仍为英语。
- 加载失败时回退到基础内置词库。

### 暂不做

- 玩家自定义词库导入。
- 原生日文假名输入。
- URL 远程词库加载。

### 验收标准

- 网页发布版本不导入任何外部文件也能正常游玩。
- 英语旧词库兼容。
- 日文词库以“显示日文 + 输入罗马音”的方式可玩。

---

## v3.5：自定义词库导入与本地保存

### 目标

让玩家可以在网页中导入自己的 JSON 词库，并在本地保存。

### 主要功能

- 文件选择器导入 `.json` 词库。
- 词库格式校验。
- 保存到 `localStorage`，大词库后续再考虑 `IndexedDB`。
- 删除已导入词库。
- 导出词库。
- 词库错误提示。

### 验收标准

- 导入错误不会破坏游戏。
- 清空导入词库后仍能回到内置词库。
- 自定义词库不作为网页发布的必要条件。

---

## v3.6：内容包与长期扩展预研

### 目标

为自定义 NPC、故事包、词库包统一导入/导出做准备。

### 主要功能

- 设计内容包结构：`characters / gifts / stories / relationships / wordPacks`。
- 冲突检测：ID、角色引用、礼物引用、故事条件可达性。
- 角色搜索、筛选、排序预研。
- 关系故事界面预研。

### 验收标准

- 有明确数据结构文档。
- 不急于做完整编辑器，先保证结构可扩展。

---

## 3. 当前推荐下一步

当前 **v3.1：版本整理与系统基线** 已基本完成。

推荐下一步优先做：

```text
v3.1-automation：归档与检查自动化
```

完成自动化后，再进入：


```text
v3.2 金币消耗与羁绊经济循环
→ v3.3 故事条件网络
→ v3.4 内置多语言词库包
→ v3.5 自定义词库导入
```

## 4. 范围控制原则

- 不在 v3.1 直接改大功能。
- 金币系统优先于多语言系统。
- 内置词库优先于导入词库。
- 日文先做罗马音输入，不做原生 IME 假名输入。
- 所有故事条件必须有玩家可理解的线索。
- 用户不需要记住复杂交接流程；后续模型应从 `START_HERE.md` 自行读取上下文。
- 每轮迭代前后都要更新 `CHANGELOG.md` 与归档说明。
