# AI 协作交接入口

> 如果你刚接手本项目，且不知道从哪里开始，请先阅读项目根目录的 `START_HERE.md`。  
> `START_HERE.md` 是给用户和后续模型使用的唯一记忆入口；本文件负责提供更详细的协作规则。

本文件是后续 AI 模型、文本协作者或代码协作者接手 `witch-shop-game` 时的详细协作说明。目标是减少重复摸索，避免跨模型协作时破坏既有结构。

## 1. 接手前必读顺序

1. `START_HERE.md`：唯一入口，用户只需要记住这个文件。
2. `AI_HANDOFF.md`：当前交接规则与协作规范。
3. `FRAMEWORK.md`：项目模块、调用链、存档与核心系统说明。
4. `docs/V3_ROADMAP.md`：v3.0 阶段路线图与小版本边界。
5. `docs/VERSION_ARCHIVE.md`：版本归档、回退与多模型协作规范。
6. `docs/DATA_CONTRACTS.md`：数据字段、ID、存档兼容契约。
7. `docs/QA_CHECKLIST.md`：修改后的手动验证清单。
8. 如修改故事、角色、礼物、羁绊或内容包，必须阅读 `STORY_SYSTEM_ROADMAP.md`。

当前已开始 v3.3.1：未解锁故事与羁绊页“下个故事”目标摘要先从后台条件清单，改为玩家可理解的半隐藏线索表达。v3.2 的礼物部分仍定位为“首批商品样品赠礼最小闭环”：并非所有商品都已进入礼物库，也不追求每个商品拥有 NPC 专属反应。完整礼物库扩充、角色偏好矩阵、礼物触发故事、特殊 NPC 专属订单链、`requirements` 统一、流失订单叙事入口、经营 IF 网络与复杂身份助力均属于 v3.3 或更后续工作。如未来推进 v3.4 多语言词库包，可从英文归档恢复/拆分资源。

核心目标是将游戏从：

```text
扁平经营 + 解锁后阅读故事
```

升级为：

```text
订单表现 → 金币产出/消耗 → 商品解锁/礼物/店铺投资 → 羁绊变化
→ 特殊 NPC 访问反馈/故事解锁/回馈助力 → 新订单/新反馈
```

后续优先方向：

```text
v3.2 商品赠礼与特殊 NPC 关系循环
v3.3 故事条件网络、特殊委托与半隐藏预告
v3.4 内置多语言词库包系统
v3.5 自定义词库导入
```

## 3. 修改前流程

1. 明确本轮目标、范围和成功标准。
2. 判断属于小改动还是大改动。
3. 创建归档：
   - 小改动：备份受影响文件到 `archives/<version>/files/`。
   - 大改动或基线：创建完整可运行快照 `archives/<version>-full/`。
4. 如涉及数据结构，先检查 `docs/DATA_CONTRACTS.md`。
5. 如涉及故事、角色、礼物、羁绊，先检查 `STORY_SYSTEM_ROADMAP.md`。

## 4. 修改后流程

修改完成后至少执行：

```bash
node tools/project_check.js
```

如果本轮只修改运行时代码且想快速定位语法问题，可额外或单独运行：

```bash
node tools/check_syntax.js
```

并根据改动类型参考：

```text
docs/QA_CHECKLIST.md
```

完成后更新：

1. `CHANGELOG.md`
2. 本轮归档目录中的 `MANIFEST.md`

## 5. 不要轻易做的事

- 不要无说明地改 `localStorage` 存档 key：`witchShopSave_v1`。
- 不要直接删除旧字段，例如故事旧字段 `unlockPhase / unlockCoins`。
- 不要重命名已有核心 ID，例如模块 ID、升级 ID、角色 ID。
- 不要删除历史归档目录。
- 不要把大型功能混进文档整理任务。
- 不要让玩家可见线索直接暴露后台字段，例如 `giftId = xxx`。

## 6. 当前推荐下一步

当前建议继续推进：

```text
v3.3：故事条件网络、特殊委托与半隐藏预告
```

封版 QA 已在 `docs/QA_CHECKLIST.md` 中补充“v3.2.4 封版端到端场景”。若需要回看 v3.2 是否可封版，应优先按该段手动检查新存档完整闭环与旧存档兼容，而不是继续扩张 v3.2 新功能。

优先确认已完成的最小闭环：

```text
商品解锁介绍 → 样品库存 → 商品赠礼 → 羁绊提升
→ 熟客来访台词变化 → 羁绊阶段回馈提示
```

当前 v3.3.1 已先处理：

```text
半隐藏故事预告统一：让未解锁故事显示玩家可理解的线索，不暴露 giftId、unlockStats 等后台字段；羁绊页目标摘要同步使用可读线索。
```

v3.3 优先解决：

1. 在 v3.3.1 基础上继续统一 `requirements` 字段结构，避免后续故事条件继续分散在旧字段中。
2. 流失订单叙事入口：让 `customersLost`、异常订单或失败经营成为故事线索，而不是单纯惩罚。
3. 经营 IF / 里世界入口最小闭环：用 `shopTraits`、`coinSpendStats`、商品赠礼、羁绊与熟客来访记录解锁少量店铺线 / 里世界故事。
4. 礼物库与故事联动扩充：把未进入 `giftable` 的商品按模块、标签和角色故事分批接入；为重点角色设计偏好矩阵、专属反应或礼物触发故事，避免在 v3.2 封版阶段只做无叙事支撑的商品清单扩张。

暂不建议继续在 v3.2 中扩张完整特殊委托链、复杂数值助力或大规模新故事条件字段。
